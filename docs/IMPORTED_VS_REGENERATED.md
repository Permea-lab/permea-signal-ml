# Imported vs Regenerated

## Purpose

This document defines the distinction between imported legacy artifacts and regenerated benchmark-contract artifacts in `permea-signal-ml`.

The distinction matters because legacy outputs may still be useful evidence, but they should not automatically be treated as current benchmark results.

## Imported legacy artifacts

An imported legacy artifact is a file produced outside the current `permea-signal-ml` benchmark contract and brought into the repository for reference, comparison, or historical continuity.

Typical imported legacy artifacts include:

- `rf_feature_importance.png`
- `baseline_ml_results.csv`
- `benchmark_ml_results.csv`
- `dataset_with_features.csv`

Imported legacy artifacts may have real value, but they usually have one or more limitations:

- incomplete provenance under the current repository contract
- no current run manifest
- no current naming convention
- no current summary or prediction export contract
- partial or missing dataset versioning, attribution, or licensing confirmation

## Regenerated benchmark artifacts

A regenerated benchmark artifact is a file produced by rerunning the workflow inside `permea-signal-ml` under the current benchmark contract.

These artifacts should be tied to:

- an explicit dataset config
- an explicit feature config
- an explicit model config
- an explicit evaluation config
- stable output naming
- a run manifest

Typical regenerated benchmark artifacts include:

- metrics JSON
- predictions CSV
- ranking CSV
- summary CSV
- manifest JSON
- figure outputs generated during the rerun

## Labeling rules

### Filenames

Imported legacy artifacts should keep their original names only when necessary for traceability, but they should be placed and documented as legacy imports.

Regenerated artifacts should follow current repository naming rules such as:

- `<run_id>_metrics.json`
- `<run_id>_predictions.csv`
- `<run_id>_ranking.csv`
- `<run_id>_summary.csv`
- `<run_id>_manifest.json`
- `<run_id>_score_distribution.png`

### Captions and notes

Imported legacy artifacts should be labeled explicitly as imported legacy material.

Examples:

- `Imported legacy figure from earlier BBB analysis; not rerun under current benchmark contract.`
- `Imported legacy metrics table; useful for reference but not a current benchmark-contract output.`

Regenerated artifacts should be labeled as current benchmark-contract outputs.

Examples:

- `Regenerated under the current benchmark contract using explicit configs and manifest tracking.`
- `Current benchmark-contract rerun output.`

### Documentation

Repository documentation should distinguish clearly between:

- imported for historical reference
- regenerated for current benchmark evidence

This distinction should appear in migration notes, method notes, captions, and any publication-support packaging.

## What should never be presented as current benchmark evidence unless rerun

The following should not be presented as current benchmark evidence unless rerun under the current contract:

- `rf_feature_importance.png`
- `baseline_ml_results.csv`
- `benchmark_ml_results.csv`
- `dataset_with_features.csv` when used only as an imported legacy processed dataset
- any legacy figure without a current run manifest
- any legacy result summary without current config and split-policy traceability

## Example interpretation

- `rf_feature_importance.png`
  - imported legacy artifact if copied from earlier BBB analysis
  - regenerated benchmark artifact only if reproduced by a current rerun

- `baseline_ml_results.csv`
  - imported legacy artifact if copied from earlier BBB analysis
  - not sufficient by itself as current benchmark evidence

- `benchmark_ml_results.csv`
  - imported legacy artifact if copied from earlier BBB analysis
  - useful reference for expected baseline behavior
  - still not current benchmark evidence without rerun

- `dataset_with_features.csv`
  - imported legacy processed dataset candidate for onboarding
  - not by itself a benchmark run artifact

- future regenerated predictions, manifest, and summary outputs
  - current benchmark-contract artifacts if produced by the repository pipeline after onboarding and rerun

## Practical rule

If an artifact was not generated by the current repository workflow with explicit configs and a manifest, treat it as imported legacy material rather than current benchmark evidence.
